# Debugging Runbook — OpenMuara

How to inspect OpenMuara state, replay webhooks, and resolve common problems.

---

## Inspect current state

### CLI quick checks

```bash
muara doctor
muara audit list --limit 50
muara transaction list --limit 50
muara transaction inspect <ref>
muara webhook list --limit 50
```

### Admin dashboard

Open `http://127.0.0.1:9000/_admin` in a browser. The dashboard shows:

- Enabled providers
- Recent transactions with status and trace ID
- Recent webhook attempts with trace ID
- Replay controls

### API endpoints

```bash
# Health
curl -s http://127.0.0.1:9000/healthz
curl -s http://127.0.0.1:9000/readyz

# Transactions (paginated)
curl -s 'http://127.0.0.1:9000/_admin/transactions?limit=50&offset=0'

# Single transaction
curl -s 'http://127.0.0.1:9000/_admin/transactions/<ref>'

# Webhook attempts (paginated)
curl -s 'http://127.0.0.1:9000/_admin/webhooks?limit=50&offset=0'

# Single webhook attempt
curl -s 'http://127.0.0.1:9000/_admin/webhooks/<ref>'

# Audit log (paginated)
curl -s 'http://127.0.0.1:9000/_admin/audit?limit=50&offset=0'

# Metrics
curl -s http://127.0.0.1:9000/metrics
```

## Replay a webhook

If a webhook failed to reach your consumer, fix the consumer first, then replay the attempt.

### From the CLI

```bash
muara webhook list --status failed
muara webhook replay <attempt-id>
```

### From the dashboard

1. Open `/_admin`.
2. Find the failed attempt in the webhooks table.
3. Click the **Replay** button for that row.

Replaying re-uses the stored payload and signature but creates a new delivery attempt.

## Check the audit log

Audit events capture security-relevant actions with an actor, resource, and result.

```bash
muara audit list --limit 100 --since 2026-06-01T00:00:00Z
```

Useful for answering:

- Who or what initiated a charge/refund/scenario change?
- Did a provider initialize successfully?
- Was a webhook delivered or did it fail?

## Common problems

### Webhook not delivered

1. Verify `webhook.url` is set in `.muara/config.yml`.
2. Confirm the URL is reachable from OpenMuara:
   ```bash
   curl -X POST <webhook-url> -d '{}'
   ```
3. Find the transaction's `trace_id`:
   ```bash
   muara transaction inspect <ref>
   ```
4. Check that the outgoing webhook includes the `X-Trace-Id` header in the webhook attempt details.
5. Inspect attempts: `muara webhook list --status failed`.
6. Verify the consumer is not returning 4xx; a 4xx response is recorded as failed.
7. Replay after fixing the consumer.

### Correlating requests with trace IDs

OpenMuara generates a `trace_id` for every incoming request and propagates it to the transaction ledger and outgoing webhooks as the `X-Trace-Id` header.

To correlate a provider request with its webhook:

1. Send the request with your own `X-Trace-Id` header, or use the one generated by OpenMuara.
2. Look up the transaction: `muara transaction inspect <ref>`.
3. Look up the webhook attempt: `muara webhook inspect <ref>`.
4. Both records carry the same `trace_id`.

### Provider signature mismatch

**Fawry:**

Ensure the signature is SHA256 of:

```
merchantCode + merchantRefNum + customerProfileId + returnUrl + itemId + quantity + price + merchantSecurityKey
```

The configured `merchant_security_key` must match the value used to sign.

**SenangPay:**

Ensure the callback MD5 is:

```
secret_key + amount + status_order_id + order_id + transaction_id
```

**Stripe:**

Outgoing Stripe webhooks include a `Stripe-Signature` header. Verify with the configured `webhook_secret`.

### CSRF or CORS errors

- CSRF errors on `/_admin/*` mutations usually mean the `openmuara_csrf` cookie is missing or the `X-CSRF-Token` header does not match.
- The dashboard reads the token from a `<meta name="csrf-token">` tag.
- `/_admin/webhook-receiver` is CSRF-exempt because it receives server-to-server webhooks.
- CORS errors from a test app usually mean the origin is not in the `cors.allowed_origins` list.

### Database locked or slow

OpenMuara uses a single shared SQLite connection. If you see `database is locked`:

1. Reduce concurrent writes from load tests.
2. Ensure the database is on a local filesystem.
3. Verify the file is writable: `ls -l .muara/data/ledger.db`.
4. Run a quick integrity check:
   ```bash
   sqlite3 .muara/data/ledger.db "PRAGMA integrity_check;"
   ```
5. Restart OpenMuara if a stale lock is suspected.

### Metrics missing in Prometheus

1. Confirm `GET /metrics` returns data.
2. Confirm Prometheus is scraping the correct host and port.
3. Check that `metrics_path` is `/metrics` in the scrape config.
4. Confirm firewall or container networking allows the scrape.

### Dashboard shows empty tables

1. Check that a provider is enabled and transactions exist: `GET /_admin/transactions`.
2. Open the browser console for JS errors.
3. Verify the dashboard is fetching paginated responses (`{ limit, offset, results }`), not plain arrays.
4. If using a custom port or host, ensure the dashboard JS is loaded from the same origin.

## Runtime profiling

Enable Go pprof endpoints by setting:

```yaml
server:
  pprof: true
```

Then restart OpenMuara. Profiles are available under `http://127.0.0.1:9000/_admin/debug/pprof/`.

Useful profiles:

```bash
# Live goroutines
curl -s http://127.0.0.1:9000/_admin/debug/pprof/goroutine?debug=1

# 30-second CPU profile
curl -s http://127.0.0.1:9000/_admin/debug/pprof/profile?seconds=30 > cpu.pprof

# Heap snapshot
curl -s http://127.0.0.1:9000/_admin/debug/pprof/heap > heap.pprof
```

Keep `pprof` disabled in production-like or shared environments; it exposes runtime internals.

## Getting more detail

Enable debug logging:

```yaml
log:
  level: debug
```

Then restart OpenMuara. Debug logs include signature verification steps, dispatcher decisions, and provider initialization details.
