# OpenMuara Stripe FPX & Card Payments — Decision Log

| ID | Decision | Status | Date | Rationale |
|----|----------|--------|------|-----------|
| D001 | Replace the OpenMuara-native `/v1/stripe/fpx/*` and `/v1/stripe/card/*` routes with Stripe's real APIs. | ✅ | 2026-06-30 | OpenMuara's goal is faithful provider emulation so developers can use the real Stripe SDK locally and switch to production by changing only the base URL and API key. Custom routes violate that goal. |
| D002 | Scope this initiative to single-charge only: `mode=payment` for Checkout, one-time PaymentIntents, inline `price_data`, no product catalog. | ✅ | 2026-06-30 | The user requested "single charge item" and "just price set" — no product catalog or subscription creation. This matches the simplest Stripe SDK usage. |
| D003 | OpenMuara will host the checkout/authentication pages at the session `url` and `next_action.redirect_to_url.url`. | ✅ | 2026-06-30 | Real Stripe returns Stripe-hosted URLs; OpenMuara emulates them locally so the same SDK redirect flow works without code changes. The existing code already sets `url` to `/v1/checkout/sessions/{id}/pay`; P01 implements the missing handler. |
| D004 | Implement both Stripe Checkout Sessions and PaymentIntents because both are common SDK entry points for FPX/card. | ✅ | 2026-06-30 | Checkout Sessions provide a hosted checkout page; PaymentIntents are used for custom payment UIs. Supporting both maximizes SDK compatibility. |
| D005 | Webhook configuration follows the Stripe Dashboard pattern: endpoint URL, enabled events, and a displayed signing secret. | ✅ | 2026-06-30 | This makes the local admin UI feel familiar to Stripe users and keeps the outbound webhook contract aligned with Stripe's `Stripe-Signature` behavior. |
| D006 | Webhook configuration changes are persisted to `.muara/config.yml` and applied to the running dispatcher without restart. | ✅ | 2026-06-30 | Local-first OpenMuara config already lives in `.muara/config.yml`; persisting there survives restarts. Updating the running dispatcher keeps the UX smooth. |
| D007 | Stripe-compatible error responses use the shape `{"error":{"type":"...","code":"...","param":"...","message":"..."}}`. | ✅ | 2026-06-30 | Matches the Stripe API so SDK clients deserialize errors correctly. |
| D008 | Test payment method tokens are accepted for PaymentIntents confirm: `pm_card_visa`, `pm_card_mastercard` for cards; `pm_fpx_maybank`, `pm_fpx_cimb`, `pm_fpx_publicbank` for FPX. | ✅ | 2026-06-30 | Gives developers deterministic test tokens that match Stripe's `pm_*` naming convention. Unknown tokens return `invalid_request_error` / `resource_missing`. |
| D009 | If provider `baseURL` is empty, local redirect/checkout URLs are derived from the incoming request host/scheme. | ✅ | 2026-06-30 | Avoids broken URLs in tests or when the user has not explicitly configured a base URL. |
| D010 | Checkout Session pay actions clear the ledger transaction idempotency key before updating so status transitions are persisted even when `client_reference_id` was used at creation. | ✅ | 2026-07-01 | The ledger's `CreateOrGet` short-circuits on idempotency keys, so updates must bypass that mapping while keeping the original reference intact. |
| D011 | Stripe webhook configuration is persisted under `providers.stripe.config.webhook_url` and `providers.stripe.config.enabled_events` in `.muara/config.yml` and applied to the running dispatcher without restart. | ✅ | 2026-07-01 | Local-first OpenMuara config already lives in `.muara/config.yml`; persisting there survives restarts. Updating the running dispatcher keeps the UX smooth. |
